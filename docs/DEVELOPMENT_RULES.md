# DEVELOPMENT_RULES

## 제품 판단 기준

새 기능을 만들 때는 아래 질문을 먼저 통과해야 한다.

1. 운영자의 반복 업무를 줄이는가?
2. 회원의 참석 행동을 더 명확하게 만드는가?
3. 노쇼와 늦은 취소를 줄이는가?
4. 회원 또는 용병의 실제 참석 여부를 신뢰할 수 있게 만드는가?
5. 회원이 본인의 참석률, 신뢰도, 기록을 자연스럽게 의식하게 만드는가?
6. 축구/풋살 팀 운영에 유용한 데이터를 남기는가?
7. 포지션, 포메이션, 경기 기록이 나중에 승률 분석으로 이어지는가?
8. 축구/풋살에 유용하지만 다른 정기 모임에도 확장 가능한가?
9. 개인정보를 최소한으로 저장하는가?

## 개발 원칙

- 모바일 우선으로 설계한다.
- MVP에서는 규칙 기반 로직을 우선한다.
- Supabase Row Level Security를 전제로 권한을 설계한다.
- 출석, 경기 기록, 권한 변경, 라인업 확정처럼 신뢰가 필요한 데이터는 변경 이력을 남긴다.
- 스포츠 종속 필드는 가능한 한 `sport_type`, `position_code`, `formation_code`처럼 확장 가능한 코드값으로 둔다.
- 팀 멤버와 경기 단위 용병은 명확히 분리한다.
- GPS 출석은 상시 추적이 아니라 경기 출석 검증 이벤트로만 저장한다.
- AI 추천 기능은 설명 가능한 근거 데이터가 쌓인 뒤 만든다.

## 역할별 권한 원칙

- Owner: 모든 팀 운영 권한
- Manager: 경기, 참석, 용병, 결과 기록 운영 권한
- Coach: 포지션, 라인업, 포메이션, 전술 기록 관리 권한
- Member: 본인 참석 신청, 출석 체크, 개인 데이터 조회와 기본 선수 정보 관리 권한
- Guest: 초대받은 경기의 제한된 정보 조회와 본인 참석 응답/체크인 권한

## MVP 우선순위

1. 인증과 팀 생성
2. 팀 멤버 초대/권한
3. 경기 생성
4. 참석 신청
5. 운영자 출석 확정
6. 팀 초대 코드
7. 경기 단위 용병 초대
8. 신뢰도 계산
9. 랭킹
10. 경기 결과와 개인 기록

## 확장 우선순위

1. GPS/QR 출석 검증
2. 포지션과 주발 정보
3. 경기별 라인업 저장
4. 포메이션 보드 공유
5. 포지션별 출전 기록
6. 포메이션별 승률 분석
7. 상대팀별 전략 기록
8. AI Coach 추천

## 데이터 설계 원칙

- 참석 신청과 실제 출석 확정은 분리한다.
- 멤버와 용병은 같은 경기 참석자로 다룰 수 있어야 하지만, 팀 권한은 다르게 둔다.
- 경기 결과는 단순 점수뿐 아니라 포메이션, 라인업, 포지션 배치와 연결되어야 한다.
- 선수의 기본 포지션과 경기별 실제 포지션은 분리한다.
- 운영자가 입력한 공식 기록과 자동 계산/AI 추천 결과는 구분한다.
- 분석용 스냅샷은 원본 이벤트를 지우지 않고 별도 테이블 또는 뷰로 만든다.

## UX 원칙

- 모바일 첫 화면에서 오늘 해야 할 행동이 보여야 한다.
- 운영자는 경기 당일 `미응답`, `대기`, `용병`, `출석 확정`, `노쇼 위험`을 빠르게 확인할 수 있어야 한다.
- 회원은 본인의 참석률과 신뢰도를 쉽게 확인해야 한다.
- 경기 참석 응답은 1~2번 안에 끝나야 한다.
- 라인업/포메이션 화면은 경기 전에 자기 위치를 빠르게 이해하는 데 집중한다.
- 전술 분석 화면은 어려운 용어보다 근거와 추천 행동을 짧게 보여준다.

## 비기능 요구사항

- 개인정보 최소 저장
- 변경 이력 보존
- 모바일 하단 내비게이션
- 느린 네트워크에서도 핵심 화면이 먼저 보이도록 구성
- 위치 정보는 출석 검증 목적과 보관 기간을 명확히 제한
- 유료 기능으로 확장 가능한 분석 데이터 구조 유지
