# ANALYSIS_FRAMEWORK

새 기능이나 변경 요청을 검토할 때 아래 관점으로 분석한다. MoIja의 우선순위는 `운영 안정화 -> 팀 데이터화 -> 경기력 향상 -> AI Coach` 순서다.

## 1. 제품 관점

- 이 기능이 노쇼 감소나 참여율 향상에 직접 연결되는가?
- 운영자의 수작업을 줄이는가?
- 멤버가 자신의 참석 상태와 책임을 더 쉽게 이해하는가?
- 회원이 아닌 용병 참석까지 자연스럽게 다루는가?
- 경기력 향상이나 승률 분석에 쓸 수 있는 데이터를 남기는가?

## 2. 사용자 관점

- Owner: 경기/출석/기록/권한을 빠르게 관리할 수 있는가?
- Manager: Owner 없이도 현장 출석, 용병, 노쇼 처리가 가능한가?
- Coach: 포지션, 라인업, 포메이션 판단에 도움이 되는가?
- Member: 참석 신청, 변경, 개인 참석률, 개인 기록 확인이 간단한가?
- Guest: 팀 가입 없이 초대받은 경기 참석 응답과 체크인이 가능한가?

## 3. 운영 관점

- 경기 당일 미응답자, 대기자, 용병, 노쇼 위험자가 한눈에 보이는가?
- 운영자가 직접 출석을 확정할 수 있는가?
- GPS/QR 체크인이 실패해도 운영자가 예외 처리할 수 있는가?
- 취소 발생 시 대기자나 용병으로 빠르게 자리를 채울 수 있는가?
- 리마인드나 알림 자동화로 반복 업무를 줄일 수 있는가?

## 4. 데이터 관점

- 출석 상태 변경 이력이 남는가?
- 참석 신청과 실제 출석 확정이 분리되는가?
- 멤버와 용병이 구분되지만 경기 참석자로 함께 분석 가능한가?
- 시즌별 통계로 집계 가능한가?
- 신뢰도 계산에 필요한 입력값이 보존되는가?
- 공식 기록, 임시 입력, AI 추천이 구분되는가?
- 포지션, 포메이션, 경기 결과가 연결되는가?

## 5. 권한 관점

- 역할별로 읽기/쓰기 권한이 분리되는가?
- 멤버가 직접 수정하면 안 되는 포지션/공식 기록이 보호되는가?
- Guest가 팀 멤버 데이터에 접근하지 못하는가?
- Coach가 라인업을 관리하되 팀 권한 전체를 갖지 않아도 되는가?
- Supabase RLS로 표현 가능한 규칙인가?

## 6. UX 관점

- 모바일 첫 화면에서 오늘 해야 할 행동이 보이는가?
- 버튼과 상태명이 명확한가?
- 참석, 불참, 대기, 노쇼, 체크인, 운영자 확정 상태가 헷갈리지 않는가?
- 초대받은 용병이 별도 설명 없이 참석 응답을 할 수 있는가?
- 라인업/작전판에서 내 위치와 역할을 빠르게 이해할 수 있는가?
- 하단 메뉴 구조와 맞는가?

## 7. 경기력 관점

- 선수의 기본 포지션과 경기별 실제 포지션을 분리해 저장하는가?
- 주발, 가능 포지션, 참석 안정성이 라인업 판단에 반영되는가?
- 포메이션별 승률과 득실 차를 계산할 수 있는가?
- 상대팀별 전략 기록을 남길 수 있는가?
- 경기 중 전술 변경 기록을 저장할 수 있는가?
- 운영자/코치가 다음 경기에서 바로 참고할 수 있는 형태인가?

## 8. 확장 관점

- 축구/풋살 전용 용어가 DB 핵심 구조에 과하게 박혀 있지 않은가?
- 러닝 크루, 스터디 모임 등으로 확장해도 출석/신뢰도 핵심 구조는 유지 가능한가?
- 포지션/전술 기능은 스포츠 모듈로 분리 가능한가?
- 유료 기능으로 분리할 수 있는 데이터 가치가 있는가?

## 9. AI 관점

- MVP에서는 규칙 기반으로 충분한가?
- 나중에 AI Assistant가 답변할 수 있도록 데이터가 구조화되는가?
- 추천/분석 결과의 근거를 사용자에게 설명할 수 있는가?
- AI가 공식 기록을 덮어쓰지 않고 제안 또는 초안으로만 남는가?
- 경기 중 추천은 어떤 입력 데이터와 한계를 기반으로 하는지 표시할 수 있는가?

## 10. 우선순위 판단

기능 요청이 들어오면 아래 순서로 분류한다.

1. 출석/노쇼/운영 신뢰도에 직접 필요한가?
2. MVP 출시에 필요한가?
3. 나중에 경기력 분석을 위해 지금부터 쌓아야 하는 데이터인가?
4. 지금은 문서와 스키마만 준비하고 UI/AI 구현은 뒤로 미뤄도 되는가?
5. 단순히 멋있어 보이는 기능이면 제외하거나 로드맵으로 보낸다.
