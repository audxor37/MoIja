# ANALYSIS_FRAMEWORK

새 기능이나 변경 요청을 검토할 때 아래 관점으로 분석한다.

## 1. 제품 관점

- 이 기능이 노쇼 감소나 참여율 향상에 직접 연결되는가?
- 운영자의 수작업을 줄이는가?
- 멤버가 자신의 상태와 책임을 더 쉽게 이해하는가?

## 2. 사용자 관점

- Owner: 경기/출석/기록을 빠르게 관리할 수 있는가?
- Manager: Owner 없이도 현장 운영이 가능한가?
- Coach: 포지션과 라인업 판단에 도움이 되는가?
- Member: 참석 신청, 변경, 개인 기록 확인이 간단한가?

## 3. 데이터 관점

- 출석 상태 변경 이력이 남는가?
- 시즌별 통계로 집계 가능한가?
- 신뢰도 계산에 필요한 입력값이 보존되는가?
- 공식 기록과 임시 입력이 구분되는가?

## 4. 권한 관점

- 역할별로 읽기/쓰기 권한이 분리되는가?
- 멤버가 직접 수정하면 안 되는 포지션/공식 기록이 보호되는가?
- Supabase RLS로 표현 가능한 규칙인가?

## 5. UX 관점

- 모바일 첫 화면에서 오늘 해야 할 행동이 보이는가?
- 버튼과 상태명이 명확한가?
- 참석, 불참, 대기, 노쇼 상태가 헷갈리지 않는가?
- 하단 메뉴 구조와 맞는가?

## 6. 확장 관점

- 축구/풋살 전용 용어가 DB 핵심 구조에 과하게 박혀 있지 않은가?
- 러닝 크루, 스터디 모임 등으로 확장해도 유지 가능한가?
- 유료 기능으로 분리할 수 있는 데이터 가치가 있는가?

## 7. AI 관점

- MVP에서는 규칙 기반으로 충분한가?
- 나중에 AI Assistant가 답변할 수 있도록 데이터가 구조화되는가?
- 추천/분석 결과의 근거를 사용자에게 설명할 수 있는가?
