# 데모 기준 MoIja 리디자인 스펙

## 배경

사용자가 제공한 데모 사이트는 MoIja를 일반 SaaS 대시보드가 아니라 모바일 스포츠 팀 운영 앱처럼 보여준다. 현재 로컬 앱은 밝은 운영 도구 톤이 강하므로, 전체 UI를 데모의 어두운 모바일 앱 디자인과 통일하되 기존 제품 원칙은 유지한다.

이 리디자인은 기능 추가보다 화면 체계 변경이 목적이다. 출석 신청과 운영자 확정 분리, 권한 구분, 모바일 핵심 행동 1~2회 접근 원칙은 변경하지 않는다.

## 목표

- 데모와 같은 어두운 모바일 앱 톤으로 전체 화면을 통일한다.
- 운영자가 경기 당일 해야 할 일을 첫 화면에서 바로 실행하게 한다.
- 멤버는 다음 경기, 내 참석 상태, 신뢰도를 즉시 이해하게 한다.
- 출석 체크는 체크인 기록과 운영자 최종 확정을 분리해서 보여준다.
- 화면별 임시 스타일을 줄이고 공통 컴포넌트로 재사용한다.

## 비목표

- DB 스키마, RLS, Supabase 함수 변경은 이번 디자인 변경 범위가 아니다.
- GPS/QR 실제 검증 로직은 구현하지 않는다. 화면에서는 MVP 상태에 맞게 수동 또는 준비 중 상태로 표현한다.
- 랭킹 계산 로직을 새로 만들지 않는다. 필요한 경우 기존 신뢰도 데이터 기반 읽기 화면으로 시작한다.

## 데모에서 가져올 디자인 언어

### 시각 톤

- 배경은 거의 검정에 가까운 네이비 계열을 기본으로 한다.
- 카드와 입력 영역은 배경보다 한 단계 밝은 딥 네이비를 사용한다.
- 핵심 액션은 라임색, 보조 액션은 인디고/블루 계열을 사용한다.
- 위험/노쇼/삭제는 레드 계열을 제한적으로 사용한다.
- 큰 라운드보다 8~16px 중심의 compact radius를 사용한다.

### 레이아웃

- 모바일 앱 프레임을 기준으로 설계한다.
- 각 화면은 `상단 바 -> 화면 제목 -> 핵심 요약 카드 -> 주요 행동 -> 리스트/세부 정보` 순서를 따른다.
- 하단 탭은 `홈 / 경기 / 랭킹 / 팀 / MY`를 기준으로 한다.
- 데스크톱에서는 모바일 화면 폭을 유지하고 보조 설명 영역을 둘 수 있으나, 핵심 경험은 모바일 화면 안에서 완결되어야 한다.

### 컴포넌트 문법

- 숫자 상태는 작은 metric card로 표시한다.
- 중요한 흐름은 전체 폭 primary button으로 제공한다.
- 설정/관리 항목은 아이콘, 제목, 보조 설명, chevron으로 구성된 action row를 사용한다.
- 참석 상태 선택은 segmented control로 표현한다.
- 사람 목록은 아바타/초성, 이름, 역할/포지션, 상태 badge, chevron 구조로 통일한다.

## 공통 컴포넌트 설계

### AppShell

모든 인증 후 화면을 감싸는 모바일 우선 레이아웃이다.

- 어두운 배경을 적용한다.
- 상단 바와 하단 탭을 포함한다.
- 데스크톱에서는 콘텐츠 최대 폭을 제한한다.
- 하단 탭 높이만큼 하단 여백을 보장한다.

### TopBar

- 좌측에는 뒤로가기 또는 브랜드를 둔다.
- 중앙/좌측 제목은 현재 화면명을 짧게 표시한다.
- 우측에는 알림 또는 설정 아이콘을 둔다.
- 동일한 라벨이 여러 버튼에 중복될 수 있으므로 접근성 이름은 화면 맥락을 포함해 구분한다.

### BottomNav

- `홈`, `경기`, `랭킹`, `팀`, `MY` 5개 탭을 기본으로 한다.
- 현재 활성 탭은 라임색과 상단 accent line으로 표시한다.
- 기존 `새 경기`는 하단 탭에서 빼고 경기 화면 상단 CTA로 이동한다.

### StatCard

- 짧은 label과 큰 숫자 value를 표시한다.
- 참석, 불참, 미정, 미응답, 응답률, 신뢰도 등에 사용한다.
- 색상은 상태 의미에 맞추되 한 화면에서 과도하게 다양해지지 않게 제한한다.

### ActionRow

- 운영자 할 일, 팀 관리, 내 정보 메뉴에 사용한다.
- 왼쪽 아이콘, 제목, 설명, 오른쪽 chevron을 갖는다.
- 행 전체가 클릭 가능해야 한다.

### PrimaryAction

- 화면에서 가장 중요한 한 가지 행동에 사용한다.
- 예: `라인업 보기`, `미응답자에게 알림 보내기`, `현재 인원 출석 확정`, `경기 만들기`.

### SegmentedControl

- 경기 상세 탭, 참석 상태, 출석 방식, 랭킹 종류에 사용한다.
- 모바일에서 가로 스크롤 없이 3~4개 옵션이 들어가도록 compact하게 설계한다.

## 화면별 적용 설계

### 홈

운영자와 멤버 공통으로 다음 경기 카드가 첫 화면 핵심이다.

- 상단에 브랜드, 알림, 팀/역할 정보를 표시한다.
- 다음 경기 카드에는 날짜, 시간, 상대팀, 장소, 참석 확정 수, 포지션/정원 상태, 참석 마감까지 남은 시간을 표시한다.
- 참석 응답률 progress bar와 `참석 / 불참 / 미정 / 미응답` StatCard를 둔다.
- 운영자는 `빠른 체크인`, `라인업 작성`을 바로 실행한다.
- 멤버는 `참석 신청/상태 변경`, `출석 체크`를 바로 실행한다.
- 신뢰도, 연속 참석, 다음 팀 일정은 compact card로 표시한다.

### 경기 목록

- 월간 캘린더 요약과 경기 리스트를 함께 보여준다.
- `경기 등록`은 화면 상단 우측 primary CTA로 둔다.
- 경기 카드는 날짜 블록, 경기명, 시간/장소, 참석 상태 요약, chevron으로 구성한다.
- 지난 경기는 별도 섹션으로 분리한다.

### 경기 만들기/수정

기존 긴 폼은 모바일 설정 화면처럼 재구성한다.

- 경기 이름, 날짜와 시간, 장소는 큰 입력 카드로 표시한다.
- 참석 마감, 정원/대기, 출석 방식은 설정 행으로 표시한다.
- MVP에서는 출석 방식 기본값을 수동 또는 운영자 승인 중심으로 둔다.
- 제출 CTA는 전체 폭 하단 primary button으로 둔다.

### 경기 상세

- 상단 요약 카드에 D-day, 일시, 경기명, 장소를 표시한다.
- 내부 탭은 `정보 / 참석 / 라인업`을 사용한다.
- 정보 탭에는 내 참석 상태 segmented control과 운영자 할 일 action row를 둔다.
- 운영자 할 일은 `참석 현황 관리`, `용병 초대하기`, `라인업과 작전판` 순서로 둔다.
- 경기 메모는 별도 compact card로 표시한다.
- 수정/취소는 보조 또는 위험 액션으로 낮은 우선순위에 둔다.

### 참석 관리

- 상단에 참석, 불참, 미정, 미응답 숫자를 StatCard로 표시한다.
- 가장 중요한 CTA는 `미응답자에게 알림 보내기`로 둔다.
- 보조 CTA는 `용병 초대`, `대기자 관리`로 둔다.
- 참가자 리스트는 초성, 이름, 역할/포지션, 최근 참석률, 현재 상태 badge를 표시한다.
- 운영자만 상태 확정/노쇼 처리를 할 수 있다.

### 출석 체크

- 출석 방식 segmented control은 `수동 / QR / GPS`를 보여주되 MVP에서는 수동을 기본 활성화한다.
- 체크인 수와 전체 대상 수를 큰 숫자로 표시한다.
- `운영자 확정 전, 체크인만 기록됩니다` 문구를 유지한다.
- 팀원, 용병, 노쇼 처리 필터를 제공한다.
- 사람 목록 우측에는 체크인 완료 여부를 원형 아이콘으로 표시한다.
- 하단에는 `현재 인원 출석 확정` primary CTA를 둔다.

### 라인업

- 참석 확정자 기준이라는 전제를 상단에 명확히 표시한다.
- formation segmented control을 제공한다.
- 작전판은 초록 필드로 표현하되 나머지 UI 토큰은 전체 앱과 동일하게 유지한다.
- 저장 버튼과 공유 버튼을 분리한다.
- 전술 메모는 compact memo card로 표시한다.

### 랭킹

- `출석왕 / 득점왕 / 도움왕 / 신뢰도` segmented control을 둔다.
- MVP에서는 실제 데이터가 있는 범위부터 표시한다.
- 상단 포디움은 동기부여용, 하단 리스트는 데이터 확인용으로 사용한다.
- 노쇼, 미응답 등 부정 지표는 낙인보다 개선 행동으로 연결되게 문구를 조정한다.

### 팀

- 팀 프로필 카드에 팀명, 짧은 슬로건, 설정 아이콘을 둔다.
- 이번 시즌 경기, 팀원, 평균 참석률을 StatCard로 표시한다.
- 팀 관리 action row는 `팀 초대 코드`, `운영진 권한`, `홈구장` 순서로 둔다.
- Owner/Manager/Coach 권한 구분은 기존 규칙을 유지한다.

### MY

- 프로필 카드에 이름, 팀, 역할, 신뢰도, 참석률, 연속 참석을 표시한다.
- 하단에는 `주 포지션`, `참석 이력`, `개인 경기 기록` action row를 둔다.
- 민감정보는 추가로 노출하지 않는다.

## 구현 순서

1. 디자인 토큰과 전역 CSS를 어두운 앱 테마로 바꾼다.
2. AppShell, TopBar, BottomNav, StatCard, ActionRow, SegmentedControl, PrimaryAction을 만든다.
3. 홈 대시보드와 하단 네비를 데모 구조로 변경한다.
4. 경기 상세과 참석 운영 화면을 데모 구조로 변경한다.
5. 경기 목록과 경기 생성/수정 화면을 변경한다.
6. 팀, MY, 랭킹 화면을 변경한다.
7. 라인업/작전판은 기존 기능을 유지하며 시각 토큰만 통일한다.

## 검증 기준

- 모바일 폭에서 가로 스크롤이 없어야 한다.
- 핵심 CTA는 각 화면에서 1개가 명확해야 한다.
- 운영자와 멤버 권한별로 보이는 행동이 섞이지 않아야 한다.
- 출석 신청, 체크인, 운영자 확정은 화면 문구와 상태에서 구분되어야 한다.
- `npm run typecheck`와 `npm run build`를 통과해야 한다.
- 빌드 전에는 Next dev/build 프로세스가 `.next`를 점유하지 않는지 확인한다.

## 구현 결정

- 랭킹 탭은 실제 라우트로 만들되, 첫 구현에서는 기존 신뢰도/출석 데이터로 표시 가능한 읽기 전용 화면부터 제공한다.
- 데스크톱에서는 데모처럼 모바일 앱 화면을 좌측 또는 중앙에 두고, 우측에는 짧은 제품 메시지 영역을 유지한다.
- QR/GPS 버튼은 출석 체크 화면에 비활성 준비 중 상태로 노출하고, 실제 출석 처리는 수동과 운영자 확정을 기본 흐름으로 둔다.
